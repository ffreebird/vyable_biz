# Vyable Biz: 컨텍스트 인터뷰 프로세스 (Logic Specification)

> **문서 목적**: 사용자의 아이디어를 비즈니스 맥락(RFP)으로 변환하는 AI 엔진의 **실행 로직**을 명세합니다.

---

## 0. 핵심 원칙 (Core Principle)

> **"질문의 목적은 정보 수집이 아니라 사고 촉발이다"**

### 파생 원칙

| # | 원칙 | 근거 (왜) | 판단 기준 |
|---|------|----------|----------|
| 1 | **사용자 결정만 묻는다** | 결정할 수 없는 걸 물으면 사고가 아니라 혼란이 생긴다 | "사용자가 결정할 수 있는가?" |
| 2 | **Yes/No가 아닌 선택을 준다** | Yes/No는 사고를 닫고, 선택지는 사고를 연다 | "사고를 확장하는가, 제한하는가?" |
| 3 | **먼저 제안하고 선택받는다** | 빈 종이는 막막함을 주고, 선택지는 생각의 출발점을 준다 | "스스로 떠올릴 수 있는가?" |
| 4 | **놓친 관점을 드러낸다** | 아는 걸 말하게 하는 건 정보 수집. 모르던 걸 깨닫게 하는 게 사고 촉발 | "새로운 걸 생각하게 되는가?" |
| 5 | **AI가 채운 건 확정 안 함** | 사용자가 생각하지 않은 건 사용자의 결정이 아니다 | AI 추론 → [추론] 태그, 확인 전까지 확정 X |
| 6 | **미정을 존중한다** | 억지로 결정하게 하면 진짜 사고가 아니다 | 모든 질문에 "아직 모르겠다" 옵션 필수 |

### 응답 상태 분류 (Response State)

| 상태 | 의미 | AI 행동 |
|------|------|---------|
| **[확정]** | 사용자가 명확히 답변 | 그대로 RFP에 반영 |
| **[추론]** | AI가 맥락에서 유추 | 확정 전까지 [추론] 태그 유지, 검증 필요 |
| **[미정]** | 사용자가 의도적으로 보류 | 존중하고 RFP 1.9에 기록 |
| **[누락]** | 질문하지 않아 정보 없음 | Gap filling 단계에서 추가 질문 |

## RFP 4단계 구조 (Tiered RFP System)

인터뷰는 4개의 단계(Phase)로 구성된다. Seed/Sprout/Tree는 사용자의 **준비 상태**에 따라 진행 여부가 결정되지만, **Final은 어떤 단계에서 끝나더라도 항상 진행**된다.

| Phase | 이름 | 범위 | 핵심 질문 | 종료 가능 |
|-------|------|------|----------|----------|
| **Seed** | 씨앗 (제품 정의) | 1~6단계 | "뭘 만드는 건가요?" | ✅ 여기서 → Final |
| **Sprout** | 새싹 (시장 검증) | 7~10단계 | "시장에서 통할까요?" | ✅ 여기서 → Final |
| **Tree** | 나무 (사업 운영) | 11~17단계 | "어떻게 굴릴 건가요?" | ✅ 여기서 → Final |
| **Final** | 마무리 (정리) | 18~20단계 | 정리 및 확정 | ❌ 항상 진행 |

### 단계 전환 원칙: 목적 질문 (Purpose Question)

> **핵심**: 각 단계가 끝나면, **다음 단계의 목적을 질문**한다.
> 사용자가 그 목적에 **"네"** → 다음 단계 진행
> 사용자가 **"아니요"/"아직이요"** → **Final로 이동** (현재까지 수집된 정보로 RFP 정리)

**Seed → Sprout 전환**
```
[6단계 완료]
AI: "이 제품을 실제 시장에서 검증하길 원하시나요?"

Case A: "네" → Sprout 진행
Case B: "아니요" / "아직이요" → Final로 이동 (Seed RFP 출력)
```

**Sprout → Tree 전환**
```
[10단계 완료]
AI: "이 서비스를 정식으로 출시하실 건가요?"

Case A: "네" → Tree 진행
Case B: "아니요" / "아직이요" → Final로 이동 (Sprout RFP 출력)
```

**Tree → Final 전환**
```
[17단계 완료]
→ 자동으로 Final 진행 (Full RFP 출력)
```

### 왜 이렇게 하는가: 목적의 자각

- "다음 단계로 갈래요?"는 확인 질문 → 금지
- "경쟁사 알아요?"는 테스트 질문 → 금지
- **다음 단계의 목적을 직접 질문**하면, 사용자가 스스로 자신의 의도를 답하게 됨
- 답변에 따라 AI가 분기 처리 → **메타인지의 외주화**
- **Final은 항상 진행**: 어느 단계에서 끝나든 수집된 정보를 정리하고 확정하는 과정은 필수

## Phase 1. 정보 수집 및 구조화 (Collection)

### 1. 서비스 정의
- **질문 (Text)**: "어떤 서비스를 만들고 싶은가요? (주관식 - 한줄)"
- **AI Internal Logic (3-Step Validation)**:
    1. **Garbage Check**: 무의미한 입력 반려.
    2. **Active Inquiry (보완)**: 입력이 너무 빈약하면, 반려하는 대신 구체적으로 되물어 정보를 받아낸다.
    3. **Pass**: 명확하면 통과.
- **Pass 조건 (Logical Seed 확립)**:
    서비스 정의에 다음 3가지 중 **2가지 이상** 포함되어야 Pass:
    | 요소 | 설명 | 예시 |
    |------|------|------|
    | [대상] | 누가 사용하는가 | "프리랜서가", "농부와 소비자가" |
    | [행위] | 무엇을 하는가 | "포트폴리오를 올리고", "거래하는" |
    | [가치] | 왜 하는가 / 어떤 결과 | "의뢰를 받는", "직거래로 신선하게" |

    **판정 예시**:
    - "쇼핑몰" → 0/3 → **Inquiry**
    - "옷 파는 쇼핑몰" → 1/3 (행위만) → **Inquiry**
    - "프리랜서가 포트폴리오 올리고 의뢰받는 플랫폼" → 3/3 → **Pass**
    - "농산물 직거래 플랫폼" → 2/3 (대상+행위 추론 가능) → **Pass**

- **Active Inquiry 규칙**:
    - **최대 2회**까지만 되묻는다.
    - 2회 후에도 부족하면 현재 상태로 진행하고, **Phase Final에서 보완**한다.
    - Inquiry 질문 형식: "**[누가]** 사용하고, **[무엇을]** 하는 서비스인가요?"
    - **"모르겠어요" 응답 시 처리**:
        | 횟수 | AI 행동 |
        |------|---------|
        | 1회차 | "어떤 문제를 해결하고 싶은지만 알려주세요" (더 쉬운 질문으로 전환) |
        | 2회차에도 모름 | 현재 상태로 진행, Phase Final에서 보완 |

- **Purpose**: AI가 이후 질문(타겟, 동작 등)을 생성하기 위한 최소한의 **'논리적 씨앗(Logical Seed)'**을 확립한다.

### 2. 타겟 정의
- **질문 (Text)**: "이 서비스를 사용해야 하는 사람은 누구인가요?"
- **선택지**:
    - [옵션 1] 한 부류의 사람들이 각자 이용합니다 (입력 1개)
        - 예: 메모앱 사용자, 게임 플레이어
    - [옵션 2] 서로 다른 역할을 가진 사람들이 연결되어 이용합니다 (입력 N개)
        - 예: 판매자-구매자, 강사-학생, 호스트-게스트
    - [옵션 3] 잘 모르겠어요
        - → AI가 서비스 유형 기반으로 역할 구조를 제안

- **AI Internal Logic**:
    1. 복잡한 관계를 묻지 않고 오직 **역할(Role)**의 구성과 명칭만 명확히 수집한다.
    2. **역할 완전성 검증**: 서비스 정의에서 추론되는 역할과 입력된 역할을 비교한다.
    3. **누락 의심 시**: 확인 질문을 던진다.

- **역할 완전성 검증 로직**:
    | 서비스 유형 | 예상 역할 | 누락 시 질문 |
    |-------------|----------|--------------|
    | 쇼핑몰/마켓플레이스 | 판매자, 구매자 | "판매하는 사람도 있나요, 본인이 직접 판매하시나요?" |
    | 콘텐츠 플랫폼 | 창작자, 소비자 | "콘텐츠를 올리는 사람과 보는 사람이 따로 있나요?" |
    | SaaS/도구 | 관리자, 일반유저 | "관리자와 일반 사용자 권한이 구분되나요?" |
    | 커뮤니티 | 운영자, 회원 | "운영자 역할이 따로 필요한가요?" |

- **운영자/본인 역할 처리**:
    - 사용자 본인이 운영자인 경우도 명시적으로 확인
    - "본인이 직접 운영하시나요?" → Yes면 [운영자] 역할 자동 추가

- **Purpose**: 서비스 내에서 상호작용할 행위자들을 식별한다.

### 2-1. 상호작용 규명
> **진입 조건**: 2단계에서 옵션 2(다중 역할)를 선택한 경우에만 진행

- **질문 (Text)**: "입력하신 역할들은 서로 어떤 관계인가요?"

- **역할 수에 따른 질문 방식**:
    | 역할 수 | 질문 방식 |
    |---------|----------|
    | 2개 | 한 번에 관계 질문: "[A]와 [B]는 어떤 관계인가요?" |
    | 3개 이상 | 쌍(pair)별로 순차 질문: "[A]와 [B]는?", "[B]와 [C]는?" |

- **관계 유형 선택지 (보조)**:
    주관식 응답이 모호할 때 다음 선택지로 명확화:
    - [A] **요청-수행**: 한쪽이 요청하고 다른 쪽이 수행 (의뢰, 주문, 예약)
    - [B] **거래-교환**: 서로 주고받음 (매매, 매칭)
    - [C] **제공-소비**: 한쪽이 제공하고 다른 쪽이 소비 (콘텐츠, 강의, 서비스)
    - [D] **협업**: 함께 무언가를 만듦 (공동 작업, 프로젝트)
    - [주관식] 직접 설명

- **AI Internal Logic**:
    1. 역할들 간의 연결 관계를 파악한다.
    2. 명확하지 않은 부분은 **[시나리오 A, B]**를 통해 확인받는다.
    3. **방향성 확인**: 단방향인지 양방향인지 파악한다.
        - "클라이언트가 디자이너에게 의뢰" → 단방향 (클라이언트 → 디자이너)
        - "판매자와 구매자가 거래" → 양방향 (판매자 ↔ 구매자)

- **3자 이상 관계 시각화** (내부 처리용):
    ```
    예: 농부, 소비자, 배송기사
    농부 ──(판매)──→ 소비자
            ↑
        배송기사(전달)
    ```
    - AI는 내부적으로 관계 그래프를 구성하여 누락된 연결을 감지한다.

- **Purpose**: 참여자들 간의 **연결 관계를 규명**한다.

### 3. 동작(Action) 도출
- **질문 (Text)**: "이 서비스에서는 어떤 동작들이 진행되어야 하나요?"

- **AI Internal Logic (Separation)**:
    - **Step A (Core)**: AI가 비즈니스 **핵심 동작** 리스트만 먼저 제안하여 선택받는다.
    - **Step B (Boilerplate)**: 로그인, 설정 등의 **뻔한 기능**은 별도 단계(또는 하단)에서 버튼식으로 빠르게 체크하게 한다. 섞지 않는다.

- **Fallback 로직 (Core 제안 불가 시)**:
    서비스 정의가 너무 추상적이어서 핵심 동작을 제안할 수 없는 경우:
    1. **주관식 우선**: "이 서비스에서 사용자가 할 수 있는 행동 3가지를 알려주세요."
    2. **카테고리 기반 제안**: 유사 서비스 카테고리를 선택받은 후 해당 기능 제안
        - "어떤 종류의 서비스에 가장 가깝나요?"
        - [A] 커머스 (사고파는) → 상품등록, 검색, 결제...
        - [B] 콘텐츠 (보여주는) → 업로드, 피드, 좋아요...
        - [C] 도구 (생산성) → 생성, 편집, 저장, 공유...
        - [D] 소셜 (연결하는) → 팔로우, 메시지, 그룹...

- **각 기능별 상태 선택**:
    AI가 제안한 기능에 대해 사용자는 다음 중 선택:
    | 선택 | 의미 | RFP 매핑 |
    |------|------|----------|
    | **[필수]** | 첫 버전에 꼭 필요 | 1.3.1 핵심 기능 |
    | **[나중에]** | 있으면 좋지만 첫 버전 아님 | 1.3.2 부가 기능 |
    | **[불필요]** | 이 서비스에 안 맞음 | 1.3.3 제외 기능 |
    | **[미정]** | 아직 모르겠음 | 1.9 결정 보류 |

- **사용자 추가 기능 처리**:
    1. "기타" 입력으로 사용자가 기능을 추가한 경우
    2. AI가 분류 질문: "추가하신 [기능명]은 첫 버전에 꼭 필요한가요?"
    3. 위 4가지 상태 중 하나로 분류

- **Purpose**: 추상적인 서비스 가치를 실제 개발 가능한 **'기능(Function)' 단위로 구체화(Materialization)**한다.

### 4. 핵심 동작(Core Domain) 선정
- **질문 (Text)**: "이 동작들 중 이 서비스의 제일 핵심이 되는 하나의 동작은 무엇인가요?"
- **선택지**: [3단계에서 [필수]로 선택한 기능들] + **[미정] 옵션**

- **진입 조건 체크 ([필수] 기능 0개 시)**:
    - [필수] 1개 이상 → 정상 진행
    - [필수] 0개:
        - "아직 첫 버전에 꼭 필요한 기능이 정해지지 않았네요."
        - "가장 중요하다고 생각하는 기능 하나만 골라볼까요?"
        - → 3단계 [미정] 기능들 다시 제시
        - → 선택 시 [필수] + Core로 지정 / [미정] 선택 시 RFP 1.9에 기록

- **AI Internal Logic**:
    1. 추가적인 가정이나 시나리오 없이 **단순 명료하게 하나만** 고르게 한다.
    2. **선택 검증 (2단계 검증)**:
        - **Step 1 (필수성 검증)**: "**[선택한 기능]** 없이도 이 서비스가 성립할 수 있나요?"
            - **Yes** → "그렇다면 진짜 없으면 안 되는 기능은 뭘까요?" (재선택 유도)
            - **No** → Step 2로
        - **Step 2 (독자성 검증)**: "다른 경쟁 서비스도 이 기능이 있나요?"
            - **Yes** → "그렇다면 이 서비스만의 특별한 점은 뭘까요?" (차별점 탐색)
            - **No** → 확정
            - **모름** → 일단 확정, Phase 2(8단계)에서 재검증
    3. **[미정] 선택 시**: RFP 1.9에 기록, Phase Final에서 재논의

- **후속 질문 (선택적)**:
    - "왜 이 기능이 가장 핵심이라고 생각하시나요? (주관식)"
    - 응답은 RFP 1.3.1 핵심 기능의 **"연결된 목표"**에 매핑

- **Purpose**: 서비스가 성립하기 위해 절대 제거할 수 없는 **'단 하나의 본질적 기능(MVP Core)'**을 식별한다.

### 5. 가치 제안 (Value Proposition)
- **질문 (Text)**: "고객들이 이 서비스를 이용해야 하는 이유는 무엇인가요? (주관식)"
- **[미정] 허용**: "아직 명확하지 않아요" 선택 가능 → RFP 1.9에 기록

- **AI Internal Logic (가치 검증)**:
    | 응답 유형 | 판정 | AI 행동 |
    |----------|------|---------|
    | 가격 기반 ("싸서", "무료라서") | ⚠️ 약한 가치 | "가격 외에 다른 이유가 있을까요?" 후속 질문 |
    | 기능 기반 ("~기능이 있어서") | 중립 | Pass, Phase 2(8번)에서 경쟁사 대비 검증 |
    | 결과 기반 ("~를 얻을 수 있어서") | ✅ 강한 가치 | Pass |

- **가격 기반 응답 반복 시 처리**:
    | 횟수 | AI 행동 |
    |------|---------|
    | 1회 | "가격 외에 다른 이유가 있을까요?" |
    | 2회 | "가격이 싼 이유가 있나요? (예: 중간 유통 제거)" → 새로운 관점 제공 |
    | 3회 | 더 이상 묻지 않음. "가격 경쟁" 기록 + RFP 1.9에 "차별화 전략 재검토 필요" 추가 |

- **8번(차별점 도출)과의 연결**:
    - 5번 응답을 8번에서 경쟁사 대비 검증
    - 경쟁사도 동일한 가치를 제공하면 → Phase Final에서 차별점 재조정

- **Purpose**: 기존 해결 방식(Alternative) 대비 우리 서비스가 제공하는 확실한 **'효용의 우위(Relative Advantage)'**를 정의한다.

### 6. 실행 환경 (Platform)
- **질문 (Text)**: "이 서비스의 실행 환경은 어딘가요?"
- **선택지**:
    - [A] 웹 (PC/모바일 브라우저)
    - [B] 모바일 앱 (iOS/Android)
    - [C] 둘 다 (웹 + 앱)
    - [D] 아직 정하지 않았어요 → **[미정]** 상태로 RFP 1.9에 기록

- **"둘 다" 선택 시 후속 질문**:
    - "첫 버전은 어디서 시작할까요?"
    - [웹 먼저] / [앱 먼저] / [동시에]
    - → 후순위 플랫폼은 RFP 1.3.2 부가 기능에 기록

- **AI Internal Logic (맥락 검증)**:
    타겟 사용자(2단계)와 플랫폼 적합성을 체크:
    | 타겟 특성 | 권장 플랫폼 | 불일치 시 질문 |
    |----------|------------|---------------|
    | 이동 중 사용 (배달, 현장 작업) | 앱 | "웹을 선택하셨는데, 이동 중에도 사용하나요?" |
    | 문서/이미지 작업 (디자인, 편집) | 웹 | "앱을 선택하셨는데, PC 작업이 필요하진 않나요?" |
    | 실시간 알림/푸시 필수 | 앱 | "실시간 알림이 중요하다면 앱이 더 적합할 수 있어요" |

- **RFP 매핑**:
    - 선택 결과 → RFP 1.5.1 플랫폼 결정
    - "둘 다" + 우선순위 → MVP 플랫폼 + RFP 1.3.2 부가 기능 (후순위)

- **Purpose**: 타겟 유저의 기술적 문해력과 습관을 고려하여, 가치 전달에 가장 효과적인 **'접점(Touchpoint)'**을 결정한다.

## Phase 2. 시장 적합성 검증 (Market Fit)

### 7. 경쟁 서비스 검색 (AI Search)
- **Action**: 웹 서치를 통해 경쟁 서비스를 검색하여 3개 제시

- **AI Internal Logic (Output Detail)**:
    - 단순 카드 형태가 중요한 게 아님.
    - 각 경쟁사의 **[주요 특징, 타겟, 차별점]**을 한눈에 파악할 수 있게 정리하여, 사용자가 "내 서비스는 어떻게 달라야 하는가"를 판단할 수 있는 근거를 제공한다.

- **검색 결과 확인 (사용자 피드백)**:
    - "이 서비스들이 경쟁 서비스가 맞나요?"
    - [A] 네, 맞아요 → 8번 진행
    - [B] 일부만 맞아요 → "어떤 게 아닌가요?" (제외 후 재검색, **최대 2회**)
        - 2회 후에도 불만족: "직접 경쟁사를 알려주세요" (주관식 전환)
    - [C] 아니요, 전혀 달라요 → "비슷한 서비스를 직접 알려주세요" (주관식)
    - [D] 경쟁 서비스가 없어요 → 8번 스킵, RFP 1.1.3에 "경쟁사 없음" 기록
    - [E] 잘 모르겠어요 → AI가 간략 비교 제공:
        - "[검색된 서비스]는 ~하는 서비스고, [사용자 서비스]는 ~입니다. 비슷한가요?"
        - 사용자 확인 후 [A]/[C]/[D] 중 선택

- **검색 실패 시 Fallback**:
    1. 카테고리 확장 검색 (예: "농산물 직거래" → "농산물 커머스")
    2. 그래도 없으면 사용자에게 직접 입력 요청
    3. 최종적으로 없으면 **"시장 개척형 서비스"**로 분류 → RFP 1.1.3에 기록

- **Purpose**: 유사 서비스의 존재를 확인시켜 "나만의 아이디어"라는 환상을 깨고, **객관적 비교의 좌표계(Reference Point)**를 설정한다.

### 8. 차별점 도출
> **진입 조건**: 7번에서 경쟁 서비스가 확인된 경우에만 진행 (7번 D 선택 시 스킵)

- **질문 (Text)**: "이 경쟁 서비스들 대신 사람들이 이 서비스를 사용해야만 하는 이유는 무엇인가요?"
- **선택지**: AI가 경쟁사 약점 분석 후 3가지 제안 + **[미정]** + 주관식

- **실행 로직**:
    - AI가 검색된 경쟁사의 약점을 분석하여, 공략 가능한 차별점 3가지를 보기로 제안.

- **5번(가치 제안)과의 관계 정리**:
    | 5번 (가치 제안) | 8번 (차별점) |
    |----------------|-------------|
    | "왜 이 서비스를 써야 하나?" | "왜 **경쟁사 대신** 이걸 써야 하나?" |
    | 일반적 효용 | 상대적 우위 |

    - **5번과 8번 응답이 동일할 때**:
        - AI: "경쟁사도 이 가치를 제공하지 않나요?"
        - 사용자 확인 후 → 진짜 차별점 재탐색 또는 "더 잘한다"로 확정
    - **"잘 모르겠어요" 응답 시**:
        - "경쟁사를 써본 적 있으세요?"
        - **Yes** → "어떤 점이 불편했나요?" (비교 질문으로 유도)
        - **No** → [추론] 태그 후 Phase Final에서 재검토

- **직접 입력 경쟁사 처리** (7번 C에서 주관식 입력 시):
    1. AI가 해당 서비스 검색 시도
    2. **검색 성공** → 약점 분석 후 차별점 제안
    3. **검색 실패** → "그 서비스의 아쉬운 점이 있나요?" (직접 약점 수집)

- **경쟁사 없음(7번 D) 시 대체 질문**:
    - "기존에 사람들이 이 문제를 어떻게 해결하고 있었나요?" (대체재 탐색)
    - 예: 엑셀, 카톡, 수작업, 기존 방식
    - 응답 → RFP 1.1.2 기존 해결책에 매핑

- **Purpose**: 경쟁사가 쉽게 모방할 수 없는 우리만의 **'구조적 경쟁 우위(Unfair Advantage)'**를 식별한다.

### 9. 최종 목표 (End Goal)
- **질문 (Text)**: "사용자는 이 서비스를 통해 최종적으로 **어떤 상태**가 되기를 바라나요?"
- **선택지**: AI가 3가지 예시 제안 + **[미정]** + 주관식

- **AI Internal Logic (Outcome 검증)**:
    | 응답 유형 | 판정 | AI 행동 |
    |----------|------|---------|
    | 행위 ("~하는 것") | ⚠️ 불완전 | "그걸 하면 사용자에게 어떤 변화가 생기나요?" |
    | 제품 특성 ("~한 제품") | ⚠️ 불완전 | "그 제품으로 사용자의 삶이 어떻게 달라지나요?" |
    | 사용자 상태 ("~해지는 것") | ✅ Pass | 그대로 수용 |

- **변환 예시**:
    | 입력 (불완전) | → 변환 (Outcome) |
    |--------------|------------------|
    | "옷 사는 것" | "자신감 있는 스타일링" |
    | "신선한 농산물" | "건강한 식탁, 안심하고 먹는 삶" |
    | "빠른 매칭" | "시간 절약, 효율적인 협업" |

- **[미정] 허용**: RFP 1.9에 기록

- **Purpose**: 서비스 사용을 통해 고객이 도달하게 될 최종적인 **'바람직한 상태(Desired State / Outcome)'**를 정의한다.

### 10. 수익 모델
- **질문 (Text)**: "이 서비스의 수익 모델은 어떻게 되나요?"
- **선택지 (보조)**:
    - [A] 구독 (월/연 정기 결제)
    - [B] 건당 결제 (사용할 때마다)
    - [C] 수수료 (거래 시 일정 비율)
    - [D] 광고
    - [E] 무료 (수익화 계획 없음)
    - [F] 아직 정하지 않았어요 → **[미정]** 상태로 RFP 1.9에 기록
    - [주관식] 직접 입력
- **Purpose**: 서비스 내에서 **'가치(Value)'와 '비용(Cost)'이 교환되는 메커니즘**을 설계하여 비즈니스의 지속 가능성을 확보한다.

## Phase 3. 사업 운영 (Operations)

### 11. 유입 채널 (Acquisition Channels)
- **질문 (Text)**: "고객이 이 서비스를 처음 알게 되는 곳은 어디일까요?"

- **선택지**:
    - [A] 검색 (구글, 네이버에서 검색해서)
    - [B] SNS (인스타, 유튜브, 틱톡 등)
    - [C] 지인 추천 (입소문, 공유)
    - [D] 광고 (온라인/오프라인)
    - [E] 아직 정하지 않았어요 → **[미정]**
    - [주관식] 직접 입력

- **후속 질문 (선택에 따른 필수 기능 연결)**:
    | 유입 경로 | AI 제안 | 질문 | 응답 RFP 매핑 |
    |----------|---------|------|--------------|
    | 검색 | SEO, 랜딩페이지 | "검색으로 들어온 사람에게 처음 보여줄 페이지가 있나요?" | 1.3.1 핵심 기능 |
    | SNS | 공유 기능, 미리보기 | "콘텐츠를 공유할 수 있어야 할까요?" | 1.3.1 또는 1.3.2 |
    | 지인 추천 | 초대 코드, 리워드 | "추천한 사람에게 혜택을 줄 계획이 있나요?" | 1.4.1 수익 모델 + 1.3.2 |
    | 광고 | 랜딩페이지, 전환 추적 | "광고 클릭 후 어떤 페이지로 이동하나요?" | 1.3.1 핵심 기능 |

- **AI Internal Logic**:
    - 선택된 유입 경로에 따라 **필요 기능을 자동 제안**
    - 사용자가 [필수]/[나중에]/[불필요] 중 선택
    - → RFP 1.3.1 또는 1.3.2에 반영

- **RFP 매핑**: 선택 결과 → RFP 1.1.4 유입 채널

- **Purpose**: 고객 획득 경로를 명확히 하여, 해당 채널에 맞는 기능(SEO, 공유, 랜딩 등)을 사전에 설계한다.

### 12. 성공 지표 (Success Metrics)
- **질문 (Text)**: "이 서비스가 잘 되고 있다는 걸 어떻게 알 수 있을까요?"

- **서비스 유형 분류 로직** (2단계 역할 구조 기반):
    | 역할 구조 | 서비스 유형 |
    |----------|-----------|
    | 단일 역할 (1개) | 도구/SaaS |
    | 2역할 (거래 관계) | 커머스/마켓플레이스 |
    | 2역할 (연결 관계) | 매칭 |
    | 2역할 (제공-소비 관계) | 콘텐츠 |

- **선택지** (AI가 서비스 유형에 따라 제안):
    | 서비스 유형 | 제안 지표 |
    |------------|----------|
    | 커머스 | 거래액, 재구매율, 전환율 |
    | 콘텐츠 | 방문자 수, 체류시간, 공유수 |
    | 매칭 | 매칭 성공률, 재이용률 |
    | 도구 | 활성 사용자(DAU), 작업 완료율 |
    | 구독 | 구독자 수, 이탈률, LTV |
    | **혼합형** | 두 유형의 지표 모두 제안 |
    - [미정] 아직 모르겠어요
    - [주관식] 직접 입력

- **후속 질문 (목표 수치)**:
    - "그 숫자가 얼마나 되면 '잘 되고 있다'고 볼 수 있을까요?"
    - [주관식] / [미정] 허용

- **AI Internal Logic**:
    - 지표가 너무 추상적이면 → "그걸 어떻게 측정할 수 있을까요?" 후속
    - 예: "사람들이 좋아하면" → "좋아한다는 걸 어떤 숫자로 확인할 수 있을까요?"

- **RFP 매핑**: 선택 결과 → RFP 1.6.1 핵심 지표

- **Purpose**: 성공을 측정 가능한 지표로 정의하여, MVP 이후 의사결정의 기준점을 확보한다.

### 13. MVP 검증 가설 (Validation Hypothesis)
- **질문 (Text)**: "첫 버전으로 확인하고 싶은 게 뭔가요?"

- **선택지** (AI가 서비스에서 추론하여 제안):
    - [A] 사람들이 진짜 이 문제를 겪는지 (문제 검증)
    - [B] 이 방식으로 해결이 되는지 (솔루션 검증)
    - [C] 돈을 낼 의향이 있는지 (지불 의향 검증)
    - [D] 계속 쓸 의향이 있는지 (리텐션 검증)
    - [E] 아직 정하지 않았어요 → **[미정]**
    - [주관식] 직접 입력

- **후속 질문 (검증 기준)**:
    - "그게 맞다는 걸 어떻게 알 수 있을까요?"
    - 예: "[A] 문제 검증" 선택 시 → "몇 명이 문제를 겪는다고 말하면 검증된 걸까요?"

- **AI Internal Logic**:
    - 가설과 12단계(성공 지표) 연결
    - 가설 검증 기준이 측정 가능한지 확인
    - 추상적이면 구체화 질문

- **RFP 매핑**: 선택 결과 → RFP 1.6.2 MVP 검증 가설

- **Purpose**: MVP의 목적을 "무엇을 검증할 것인가"로 명확히 하여, 불필요한 기능 추가를 방지한다.

### 14. 외부 연동 (External Dependencies)
> **진입 조건**: 3단계에서 결제, 로그인, 알림 등 외부 서비스가 필요한 기능이 [필수]로 선택된 경우

- **질문 (Text)**: "이 서비스에서 외부 연동이 필요한 부분이 있나요?"

- **선택지** (AI가 선택된 기능에서 추론하여 제안):
    | 연동 유형 | 예시 |
    |----------|------|
    | 결제 | 카드, 계좌이체, 간편결제 |
    | 로그인 | 소셜 로그인, 이메일, 전화번호 |
    | 알림 | 푸시, 문자, 이메일 |
    | 지도/위치 | 카카오맵, 네이버지도 |
    | 파일 저장 | 이미지, 동영상, 문서 |
    - [주관식] 직접 입력
    - [미정] 아직 정하지 않았어요

- **AI Internal Logic**:
    1. 3단계 기능 목록에서 외부 연동이 필요한 항목을 자동 감지
    2. 각 항목별로 선택지 제안 (한 번에 하나씩)
    3. **자동 감지 불가 시**: 스킵하고 Phase Final Gap filling에서 재확인

- **RFP 매핑**: 선택 결과 → RFP 1.5.2 외부 의존성

- **Purpose**: 외부 서비스 의존성을 조기에 파악하여, 기술 선택과 비용 예측의 기초 데이터를 확보한다.

### 15. 초기 데이터 (Initial Data)
- **질문 (Text)**: "서비스 시작할 때 미리 준비해야 하는 데이터가 있나요?"

- **선택지**:
    - [A] 네, 있어요 → 후속 질문: "어떤 데이터가 필요한가요?" (주관식)
    - [B] 아니요, 사용자가 직접 채워요
    - [C] 아직 모르겠어요 → **[미정]** 상태로 RFP 1.9에 기록

- **AI Internal Logic (맥락 검증)**:
    서비스 유형에 따라 초기 데이터 필요 여부 확인:
    | 서비스 유형 | 예상 초기 데이터 |
    |------------|----------------|
    | 커머스 | 상품 목록, 카테고리 |
    | 콘텐츠 | 초기 콘텐츠, 샘플 |
    | 매칭 | 공급자 풀, 카테고리 |
    | 도구 | 템플릿, 예시 |
    | **혼합형** | 두 유형의 초기 데이터 모두 제안 |

- **RFP 매핑**: 선택 결과 → RFP 1.5.4 데이터 제약

- **Purpose**: 서비스 론칭 전 준비해야 할 데이터를 식별하여, 콜드 스타트 문제를 사전에 대비한다.

### 16. 운영 비용 (Operating Costs)
- **질문 (Text)**: "이 서비스를 운영하면서 발생할 비용은 어떤 게 있을까요?"

- **선택지** (AI가 선택된 기능/플랫폼에서 추론하여 제안):
    | 비용 유형 | 발생 조건 |
    |----------|----------|
    | 서버/호스팅 | 기본 |
    | 앱스토어 수수료 | 앱 선택 시 |
    | PG 수수료 | 결제 기능 시 |
    | 스토리지 비용 | 파일 업로드 기능 시 |
    | 외부 API 비용 | 지도, SMS 등 연동 시 |
    - [미정] 아직 고려하지 않았어요
    - [주관식] 직접 입력

- **AI Internal Logic**:
    1. 6단계(플랫폼) + 14단계(외부 연동) + 3단계(기능 목록)에서 비용 요소 추출
    2. 비용 요소를 사용자에게 제시 (원칙 2 준수: Yes/No가 아닌 선택지 제공):
        - "이런 비용들이 발생할 수 있어요. 어떻게 충당할 계획인가요?"
        - **선택지**:
            - [A] 수익으로 충당 → 수익 모델(10단계)과 연결 확인
            - [B] 초기 투자금으로 충당
            - [C] 아직 생각 안 해봤어요 → **[미정]** 상태로 RFP 1.9에 기록
    3. 사용자가 인지하지 못한 비용 → 새로운 관점 제공

- **RFP 매핑**: 선택 결과 → RFP 1.4.2 운영 비용

- **Purpose**: 서비스 운영에 필요한 비용 구조를 사전에 파악하여, 수익 모델과의 정합성을 검토한다.

### 17. 규제 제약 (Regulatory Constraints)
- **질문 (Text)**: "이 서비스와 관련된 법적 규제나 인허가가 있을까요?"

- **선택지** (AI가 서비스 도메인에서 추론하여 제안):
    | 서비스 도메인 | 잠재 규제 |
    |--------------|----------|
    | 금융/결제 | 전자금융업, 자금세탁방지 |
    | 의료/건강 | 의료법, 의료기기법 |
    | 교육 | 학원법, 평생교육법 |
    | 식품 | 식품위생법 |
    | 개인정보 다량 | 개인정보보호법 |
    | **인력/구인** | 직업안정법, 직업소개소법 |
    | **부동산** | 공인중개사법 |
    | **여행/숙박** | 관광진흥법, 공유숙박법 |
    | **운송** | 여객자동차운수사업법 |
    - [A] 네, 있어요 → 후속: "어떤 규제인가요?" (주관식)
    - [B] 아니요, 해당 없어요
    - [C] 잘 모르겠어요 → **[미정]** 상태로 RFP 1.9에 기록

- **AI Internal Logic**:
    1. 서비스 도메인(1단계)에서 관련 규제 추론
    2. "이런 규제가 적용될 수 있는데, 확인해보셨나요?" 형태로 질문
    3. 사용자가 모를 경우 → [미정]으로 기록, 검토 권장

- **RFP 매핑**: 선택 결과 → RFP 1.5.3 규제 제약

- **Purpose**: 법적 제약을 조기에 인식하여, 불필요한 개발 리스크를 방지한다.

## Phase Final. 마무리 (Finalization)

### 18. 비정형 맥락 수집 (Open Input)
- **질문 (Text)**: "(선택) 그 외에 이 서비스를 이해하는데 필요한 내용이 있나요?"

- **입력 가이드 (예시 제공)**:
    - 참고하고 싶은 서비스나 앱
    - 이미 정해둔 제약 조건 (예: 수수료 10% 이하)
    - 향후 확장 계획 (예: B2B 진출)
    - 기존에 작성한 기획 문서나 메모
    - 기타 AI가 알아야 할 배경

- **AI Internal Logic (입력 처리)**:
    1. 입력 내용을 분류하여 RFP에 매핑:
        | 유형 | RFP 매핑 |
        |------|----------|
        | 참고 서비스 | 1.1.3 경쟁/대체재 |
        | 제약 조건 | 1.5 기술 제약 또는 1.4 수익 |
        | 확장 계획 | 1.3.2 부가 기능 또는 1.9 보류 |
        | 기획 문서 | 전체 맥락 참조용 |
    2. **긴 텍스트(500자 이상) 입력 시**:
        - AI가 핵심 내용을 요약
        - "이렇게 이해했는데 맞나요?" 확인 질문
        - 사용자 승인 후 RFP에 반영

### 19. 맥락 재구성 (Gap Detection & Fill)
**목적**: AI가 파악한 **논리적 공백(Gap)**에 대해 질문하여 완성도를 높임.

- **Step 1: RFP 초안 생성**
    - 수집된 정보로 RFP 스키마 1차 채움
    - 각 항목에 상태 태그 부여: [확정] / [추론] / [미정] / [누락]

- **Step 2: Gap 우선순위 분류**
    | 우선순위 | 기준 | 처리 |
    |----------|------|------|
    | **P0 (필수)** | MVP 구현에 필수 (결제 수단, 핵심 기능 상세) | 반드시 질문 |
    | **P1 (권장)** | 있으면 좋음 (부가 기능, 세부 정책) | 최대 3개까지 질문 |
    | **P2 (선택)** | 나중에 정해도 됨 | 스킵, RFP 1.9에 기록 |

- **Step 3: Gap 질문 (최대 5개)**
    - **P0 개수별 처리**:
        | P0 개수 | 처리 |
        |--------|------|
        | 0~5개 | P0 전부 질문 + P1 중 남은 슬롯만큼 |
        | 6~10개 | 상위 5개만 질문 + 나머지 "이것도 정해야 해요" 목록 제시 |
        | 10개+ | "아직 정해진 게 너무 적어요" + 핵심 3개(Core, 수익모델, 타겟)만 질문 |
    - **P0 우선순위**: Core → 수익모델 → 결제수단 → 플랫폼 → 타겟
    - 나머지 Gap은 RFP 1.9(결정 보류)에 자동 기록
    - 각 질문은 **[옵션 A vs B]** 형식으로 제시
    - **Example**: "수익 모델은 구독인데 결제 수단이 미정입니다. **[A: 카드 결제]**를 붙일까요, **[B: 계좌이체]**로 할까요?"

- **Step 4: [추론] 항목 검증**
    - AI가 맥락에서 추론한 항목을 사용자에게 제시
    - "~라고 이해했는데 맞나요?"
    - [맞아요] → [확정]으로 변경
    - [아니요] → 수정 입력 받음

- **AI Internal Logic (Enlightenment via Inquiry)**:
    - **원칙**: 가르치거나(Teaching) 경고(Warning)하지 않는다. 오직 **질문(Inquiry)**과 **옵션(Option)**으로 푼다.
    - **Logic**: 어떤 항목이 비어있어서 발생할 수 있는 **[잘못된 결과(Bad Result)]**를 AI가 내부적으로 인지한 뒤, 이를 해결할 수 있는 선택지를 제시한다.

- **종료 조건**:
    - P0 Gap이 모두 해소됨
    - 또는 사용자가 **"이 정도면 충분해요"** 선택 → 남은 Gap은 RFP 1.9에 기록

- **Purpose**: 각 답변 간의 **논리적 모순(Logic Conflict)**이나 결함(Gap)을 찾아, 이를 해소할 수 있는 구체적 대안(Option)을 통해 **정합성(Consistency)**을 확보한다.

### 20. 최종 확정 (Final Confirmation)

- **Step 1: RFP 요약 제시**
    - 전체 RFP를 섹션별로 요약하여 사용자에게 보여줌
    - 각 항목의 상태 표시: [확정] ✅ / [추론] 🔄 / [미정] ⏸️

- **Step 2: 완성도 체크**
    | 완성도 | 기준 | 확정 가능 | 안내 메시지 |
    |--------|------|----------|------------|
    | 🟢 80% 이상 | 핵심 항목 대부분 확정 | ✅ | "RFP가 완성되었습니다" |
    | 🟡 50~79% | 일부 핵심 항목 미정 | ✅ | "추가 정보가 필요한 항목이 있어요" + 미정 목록 안내 후 확정 |
    | 🟠 30~49% | 핵심 항목 다수 미정 | ✅ | "미정 항목이 많아요. 개발 시 많은 결정이 필요할 수 있어요" (리스크 경고 후 확정) |
    | 🔴 30% 미만 | 핵심 항목 대부분 미정 | ❌ | "아직 정해진 게 너무 적어요" + **필수 3개(서비스 정의, 타겟, 핵심 기능 1개) 추가 질문 후 재시도** |

- **최소 완성 요건 (30% 미만 시)**:
    다음 3개가 [확정] 상태여야 RFP 저장 가능:
    1. 서비스 정의 (1단계)
    2. 타겟 (2단계)
    3. 핵심 기능 1개 이상 (3, 4단계)

- **Step 3: 최종 확인**
    - "이대로 RFP를 확정할까요?"
    - [A] 네, 확정합니다 → Step 4로
    - [B] 수정하고 싶은 부분이 있어요 → 해당 단계로 이동
    - [C] 나중에 다시 할게요 → 임시 저장 후 종료

- **Step 4: RFP 저장**
    - 최종 RFP를 **RFP_Schema.md** 형식으로 생성
    - [미정] 항목은 **1.9 결정 보류**에 명시
    - [추론] 항목은 🔄 표시와 함께 기록 (추후 확정 필요)

- **Action**: 최종 RFP 생성 및 저장.
