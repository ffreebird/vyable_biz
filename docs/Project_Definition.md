### 1.1 문제 정의

#### 1.1.1 해결하려는 문제
- **문제**: 개발 지식이 없는 창작자가 AI 에이전트(Cursor 등)에게 코딩을 시키고 싶지만, **"맥락(Context)"을 제대로 전달하지 못해** 쓸모없는 결과물만 얻는다.
- **대상**: 1인 창작자, 비개발자 PM, 예비 창업가 (Single Actor).
- **상황**: 아이디어는 있지만, AI에게 뭐라고 지시해야 할지 몰라 막막해하거나, 두루뭉술하게 말해서 오류 투성이 코드를 받을 때.
- **미해결 시 결과**: AI 코딩 툴을 몇 번 써보다가 "역시 AI는 아직 멀었어" 하고 포기하거나, 개발자에게 비싼 돈을 주고 외주를 맡긴다.

#### 1.1.2 기존 해결책과 한계
- **기존 방식**:
    1. ChatGPT/Claude에게 "앱 만들어줘"라고 대충 말하기.
    2. 일반 메모장에 아이디어 끄적이기.
    3. 엑셀이나 노션 템플릿 사용.
- **한계**:
    - 챗봇은 사용자가 모르는 걸 되묻지 않고 대충 그럴싸한(Hallucinated) 답을 준다.
    - 정형화된 템플릿은 사용자의 "숨겨진 의도"와 "비즈니스 맥락"을 담아내지 못한다.
- **이 서비스의 차별점**: **"소크라테스 인터뷰"**. AI가 집요하게 질문하여 사용자의 무지를 일깨우고, 빈약한 맥락을 완벽한 구조(Context)로 채워준다.

#### 1.1.3 경쟁/대체재 분석
- **경쟁 서비스**: ChatPRD, PM용 AI 툴 (Chisel, Leiga).
- **유형**: 간접/직접 경쟁.
- **강점**: 템플릿 기반 빠른 문서 생성.
- **약점**: "개발 에이전트"를 위한 구조적 출력(System Prompt 등)보다는 "사람"을 위한 문서에 치중됨.
- **이 서비스의 포지션**: **"AI를 위한 AI" (For AI Agents)**. 사람이 아닌 기계가 이해하기 가장 완벽한 형태의 지시서를 만든다.
- **핵심 차별점**: 사용자의 아이디어를 단순 기록하는 게 아니라, **"개발 가능한 상태"로 교육(Enlighten)하고 구조화**한다.

#### 1.1.4 유입 채널 (Channels)
- **유입 경로**: 검색(SEO) 및 **AI 검색 최적화(GEO)**. "진단(Lite)" 기능을 무료 미끼로 활용하여 유입 유도.
- **필수 대응 기능**:
    - SEO/GEO 최적화된 구조화된 데이터 마크업.
    - 무료 진단 결과 페이지 공유 기능.

### 1.2 사용자 정의

#### 1.2.1 사용자 유형
- **유형**: **The Creator (창작자)**
- **정의**: 개발 능력은 부족하지만, 자신의 아이디어를 소프트웨어로 실현하고자 하는 강력한 의지가 있는 개인.
- **목적**: "내 머릿속의 생각을 실제 돌아가는 앱으로 만들기 위해" 가장 완벽한 설계도가 필요해서 사용함.

#### 1.2.2 사용자별 핵심 목표
- **유형**: The Creator
- **목적**: 실행 가능한 완벽한 PRD(Recipe) 획득.
- **성공 조건**: 여기서 만든 파일을 Cursor에 붙여넣었을 때, **질문이나 에러 없이 한 번에** 코드가 돌아갈 때.
- **방해 요소**: 사용자 본인의 모호함("그냥 알아서 해줘"), 비즈니스 논리적 모순.

### 1.3 기능 범위

#### 1.3.1 핵심 기능 (Must)
- **기능 1: Context Interview Engine**
    - 설명: 사용자의 의도를 파악하고 모순을 지적하며 구체화하는 채팅 형 인터페이스.
    - 연결된 목표: 모호함 제거 및 맥락 확보.
    - 없으면: 그냥 챗지피티랑 다를 게 없음.
- **기능 2: Recipe Generation**
    - 설명: 인터뷰 내용을 바탕으로 구조화된 PRD와 프롬프트 세트를 생성 및 다운로드.
    - 연결된 목표: 실행 가능한 결과물 획득.
    - 없으면: 상담만 하고 결과물이 없음.

#### 1.3.2 부가 기능 (Nice to have)
- **기능**: **사용자 경험 개선 (UX Optimization)**.
- **기대 효과**: 사용자가 도구를 더 직관적으로 잘 쓰게 됨. (기능 확장이 아닌 사용성 심화)
- **우선순위**: 높음.

#### 1.3.3 제외 기능 (Out of scope)
- **기능**: 직접 코딩/배포 기능.
- **제외 이유**: 이 서비스는 "설계도"를 만드는 곳이지 "공장"이 아님. 코딩은 Cursor 등 전문 툴에 맡김.
- **향후**: 영구 제외 (Role 구분 명확화).

### 1.4 수익/비용 구조

#### 1.4.1 수익 모델
- **수익원**: (장기적) '심층 진단' 및 'Recipe 생성' 유료화.
- **지불자**: 창작자 본인.
- **방식**: (현재) **내부 POC 단계이므로 과금 없음**. (추후 구독/건당 결제 도입)
- **무료/유료 경계**: 진단(Lite) 무료 / 심층(Pro) 유료 전략 예정.

#### 1.4.2 운영 비용 구조
- **비용 항목**:
    - LLM API 비용 (Gemini 3 Flash).
    - Hosting/DB.
- **월 비용 상한**: POC 단계이므로 제한 없음.

### 1.5 기술 제약

#### 1.5.1 플랫폼 결정
- **플랫폼**: **Web Only**.
- **상세**: PC 데스크탑 뷰 중심.
- **선택 이유**: 코드 에디터와 병행 사용 시나리오.

#### 1.5.2 외부 의존성
- **외부 시스템**: Google Gemini API.
- **목적**: 인터뷰 지능 및 문서 생성 핵심 엔진.
- **연동 방식**: API Key.
- **대체 가능**: 아니오.

#### 1.5.3 규제 제약
- **법규**: 개인정보보호법.
- **영향**: 회원 정보 및 대화 내용 저장 이슈.
- **필요 조치**: **[POC 단계] 일단 무시하고 기능 구현 집중**. (추후 정석 대응 예정)

#### 1.5.4 데이터 제약
- **필요 데이터**: 없음 (사용자의 머릿속이 데이터임).

### 1.6 성공 정의

#### 1.6.1 핵심 지표
- **지표**: **Zero-Shot Execution Rate**.
- **정의**: 생성된 Recipe를 AI 코딩 툴에 넣었을 때, **추가 질문이나 수정 없이** 한 번에 작동 가능한 코드가 나오는 비율.
- **목표**: 내부 테스트 시 100% 지향.
- **시점**: POC 종료 시점.

#### 1.6.2 MVP 검증 가설
- **가설**: "AI가 만든 Recipe가 완벽하다면, 개발 지식이 전혀 없는 사람도 결과물을 만들어낼 수 있을 것이다."
- **참이면**: 자동화 도구(개발 에이전트)와의 완벽한 파이프라인 완성.
- **거짓이면 (이탈하면)**: Recipe의 구조(System Prompt)를 뜯어고쳐야 함.

### 1.7 용어 정의

- **용어: Context (맥락)**
    - 정의: 기능 명세뿐만 아니라 의도, 배경, 제약사항이 포함된 정보의 총체.
- **용어: Recipe (레시피)**
    - 정의: 최종 산출물. AI Agent가 요리(코딩)할 수 있게 정리된 지시서.
- **용어: Enlightenment (일깨움)**
    - 정의: 사용자의 모순된 요구사항을 AI가 논리적으로 지적하고 올바른 방향을 제시하는 행위.
